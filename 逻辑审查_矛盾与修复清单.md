# 逻辑审查：自相矛盾 & 不符合产品逻辑之处

审查范围：`sessionStore.ts`（状态机）、`Annotation.tsx`、`TaskDetail.tsx`、`Audit.tsx`、`CReview.tsx`、`AssignModal.tsx`、`Home.tsx`、`caseSets.ts` / `summary.ts`。

产品模型基线（本轮迭代确认）：
- 分配只分 **A**；A 评完进入 QC 可抽样。
- **有人二次点 Review 并提交（B submitted）→ Double-blind**；没人二次评 → **Normal**。两者都进 QC。
- **C 覆盖即为最终**（Final Result Ready）。
- 判定链：进 QC 门槛 = A 已提交；双盲 vs Normal = B 是否已提交；已定案 = C overwrite。

## 一、严重矛盾（会导致数据错乱 / 违反产品规则）

### 1. A-submit 仍按「预配置 back-to-back」分叉，与「只分 A」新模型冲突
- 位置：[`sessionStore.ts` submitAnnotation role="A"](file:///Users/bytedance/Documents/trae_projects/draft/src/store/sessionStore.ts#L395-L410)
- 现状：A 提交时用 `currentFlow?.backToBackEnabled` 决定
  - `status`：`"A Annotation Submitted · Waiting B"` vs `"Completed Annotation"`
  - `annotationMode`：`"Back-to-Back"` vs `"Single Annotation"`
  - `currentState`：`"Back-to-Back"` vs `"Ready for C Sampling"`
- 矛盾：`AssignModal` 已删除「AB 两轮」，分配时 `backToBackEnabled` **恒为 false**。所以这条分支永远只走 else，`"Back-to-Back" / "Waiting B"` 这套状态**再也不会由分配触发**，成了死代码；但同时截图里还出现过 `"A Annotation Submitted · Waiting B"` 状态，说明存在**遗留缓存/旧路径**。双盲现在唯一来源应是「B 实际提交」。
- 建议：A 提交统一置 `status: "Completed Annotation"`、`annotationMode: "Single Annotation"`、`currentState: "Ready for C Sampling"`，不再读 `backToBackEnabled`。双盲只由 role="B" 提交时产生。

### 2. 无 role 打开标注页，提交默认写成 A，可覆盖已完成结果
- 位置：[`TaskDetail.tsx` 点击 sessionId](file:///Users/bytedance/Documents/trae_projects/draft/src/pages/TaskDetail.tsx#L209) → `/annotate/:id`（无 `?role=`）；[`Annotation.tsx` doSubmit](file:///Users/bytedance/Documents/trae_projects/draft/src/pages/Annotation.tsx#L185) 用 `reviewRole ?? "A"`。
- 矛盾：任何人点 sessionId 链接进入，都会以 **A** 身份提交，可覆盖原 A 的结果、把已进 QC 甚至已 Final 的 case 状态重置回 A。既绕过了 `canDoB`（Review）判定，也绕过了防自审。
- 建议：
  - 无 role 时进入**只读查看**模式（不允许提交），或
  - 根据当前 flow 推断合法角色（A 未评→A；A 已评且非本人→B；否则只读），拒绝非法的默认 A 提交。

### 3. 防自审文案与判定不一致（B 自审已拦但文案只说 C）
- 位置：[`Annotation.tsx` selfReviewBlocked 文案](file:///Users/bytedance/Documents/trae_projects/draft/src/pages/Annotation.tsx#L103-L112)
- 矛盾：逻辑已拦「B 不能 review 自己 A 评过的」，但提示语只写「You annotated this session as A/B, so you cannot review it as **C**」。B 被拦时看到的是错误文案。
- 建议：文案改为按 `reviewRole` 动态显示（B/C 分别说明）。

## 二、判定口径不统一（同一概念多处用不同字段）

### 4. 「双盲」判定字段散乱：`backToBackEnabled` / `annotationMode` / `bResultStatus` 混用
- 现状：
  - `Audit.tsx`、`CReview.tsx` 已统一用 `bResultStatus === "Submitted"` 判定双盲 ✅
  - 但 `sessionStore` A-submit 分支仍用 `backToBackEnabled`（见问题 1）
  - `submitAnnotation` role="A" 还会把 `annotationMode` 写成 `"Back-to-Back"`（当预配置时），与「B 未提交=Normal」冲突
- 建议：全项目**单一事实来源** = `bResultStatus === "Submitted"`。`annotationMode` / `backToBackEnabled` 要么彻底移除，要么只作为「B 已提交」的镜像，避免三份状态各写各的。

### 5. 「完成/进 QC」门槛已统一，但状态文案仍暗示需要 B
- 位置：`status` 文案 `"A Annotation Submitted · Waiting B"`
- 矛盾：QC 门槛已改为「A 提交即可」（`aResultStatus === "Submitted"`），不再等 B。但状态文案还写 "Waiting B"，会让用户以为必须等 B。
- 建议：A 提交后统一显示 `"Completed Annotation"`（或 `"Ready for QC"`），去掉 "Waiting B"。

## 三、次要 / 体验问题

### 6. C 预填种子在「B 未提交」时仍会取 aResult，但 CReview 已判为 Normal
- 位置：[`Annotation.tsx` seedActor](file:///Users/bytedance/Documents/trae_projects/draft/src/pages/Annotation.tsx#L59-L63)
- 说明：逻辑本身没错（Normal 时 C 基于 A 覆盖），但注释写「seed from B (double-blind) or A (sampling)」，与现在「Normal = 没人二评」的语义描述不完全对齐。建议更新注释，避免后续误解。

### 7. Home 的 QC Accuracy 恒为静态 `0.0%`，与「已有 C 定案」不一致
- 位置：[`Home.tsx` qcAccuracy 单元格](file:///Users/bytedance/Documents/trae_projects/draft/src/pages/Home.tsx#L208) 读 `caseSets[].qcAccuracy`（静态 "0.0%"），而 SQS/UES 等已改为 store 实时聚合。
- 矛盾：当已有 case 走完 C QC，QC Accuracy 仍显示 0.0%，与其它实时列不一致。
- 建议：要么按「C 覆盖前后一致率」实时计算，要么明确标注「QC Accuracy 需参考标注，暂不可算」并置灰，避免误导。

### 8. `caseSets.ts` / `summary.ts` 静态聚合字段已基本被 Home 实时值取代，存在死数据
- 位置：`caseSets[].sqsAvg/uesAvg/sqsPassRate/userSatisfactionAvg`、`summary.ts` 整体。
- 说明：Home 表格已改用 `taskStats` 实时值，这些静态字段仅 `qcAccuracy` 还在用。属于冗余/易误导的死数据。
- 建议：删除未使用的静态聚合字段（或整体从 store 派生），减少「两份数据不一致」的隐患。

### 9. `AssignConfig` 仍保留 B 相关字段，但 UI 已不产出
- 位置：`sessionStore.ts` `AssignConfig`（`bEmail` / `bDistribution` / `assignBothRounds` / `backToBackEnabled`）与 `distributeTaskCases` 的 back-to-back 分支。
- 说明：`AssignModal` 恒传 `false`，这些字段与分支已成死代码。
- 建议：清理 `AssignConfig` 的 B 字段和 `distributeTaskCases` 的 B 分支，只保留 A 分配。

### 10. 批量选择依赖 `s.bot`，但语义已从「A 结果」漂移
- 位置：`TaskDetail.tsx` 复选框 `disabled={!bot}`、`batchEditReasons` 只改 `s.bot`。
- 说明：`s.bot` 现在既可能是 A 的结果、也可能被 C overwrite 后的最终结果覆盖（submit role="C" 也写 `scorePatch` 到 `bot`）。Batch Edit Reasons 改的到底是哪一版，语义已模糊。
- 建议：明确 `bot` 字段代表「当前展示的权威结果」，并在批量编辑时说明作用对象；或区分 A 原始结果与最终结果两个字段。

## 五、第二轮审查新发现（含已修）

### 11. 重新 Assign QA 会把已进 QC / 已定案的 case 打回 A 阶段（已修 ✅）
- 位置：`sessionStore.ts` `assignSingleQa`
- 原问题：无论 case 处于什么状态，`assignSingleQa` 都无条件 `patchFlow({ currentState: "A Annotation", aAssignee: qaName })`。对一个已 `Ready for C Sampling` / `In C QC` / `Final Result Ready` 的 case 重新指派，会把流程**强行退回 A 阶段**，且 `aAssignee` 被改成新人而 `aResult` 仍是旧人的 —— 数据错位。
- 修复：仅当 case 仍在 A 阶段（无 flow 或 `aResultStatus !== "Submitted"`）才初始化/重置 A 流程；A 已提交后重新指派只改 `qaOwner` 标签，不动流程和 `aAnnotator`。

### 12. 重新抽样会把「上一批已抽中但没做 C」的 case 退回未抽样（已修 ✅）
- 位置：`sessionStore.ts` `startSampling`
- 原问题：每次抽样对本次未选中的 case 强制 `sampledForQC:false / currentState:"Ready for C Sampling" / sampleBatchLabel:undefined`。导致第二次抽样（例如换一个 QA 抽）会把第一批里没被重选的 case「退回」，与「抽样是相互独立的批次、可累加」的产品预期矛盾。
- 修复：抽样改为**增量**——只提升本次新选中的 case；已抽样的 case 不再被 unsample；且 `eligible` 排除已抽样的 case，新批次只从未抽样池里抽。

### 13. QA 分配显示错人（已在上一轮修 ✅）
- 见对话记录：`distributeTaskCases` 曾把 `qaOwner` 设为 operator，导致「Assign QA」列全显示分配者。已改为按各 case 实际的 A 标注人显示。

## 六、第二轮审查——观察项（暂未改，待产品确认）

### O1. Batch Edit Reasons 可改到「已被 C 定案」的最终结果
- `batchEditReasons` 只按 `s.bot` 存在就允许改，而 C overwrite 后 `s.bot` = C 的最终结果。于是批量改 reason 会**静默修改已定案 case 的最终 reason**，绕过 C 定案的不可变性。
- 建议：批量编辑排除 `Final Result Ready` 的 case，或明确允许并留痕。

### O2. `by_qa` 抽样只按 A 匹配，忽略 B
- `startSampling` 的 `by_qa` 用 `aAnnotator ?? aAssignee` 匹配。一个 case 若由 B 二次评过，按 B 抽样时抽不到。
- 建议：若产品需要「按任一参与者抽样」，匹配条件应含 B。

### O3. 同一 case 可被多个非 A 用户同时进入 Review，后提交的 B 覆盖前者
- `canDoB` 在渲染时判断 `!bAlreadyDone`，无并发锁。两人同时打开 Review 都能提交，后者覆盖前者的 B 结果。
- 说明：mock/单机场景影响有限；如上线需服务端加乐观锁或「认领」机制。
