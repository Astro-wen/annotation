# 角色可见性与权限 Plan（盲检 / 拉齐 / QC）

## 一、要解决的问题（对应三张截图）

1. **A 看到的东西太多**：vendor.a 视角能展开看到 B 行、B 的分数，甚至 C 行。盲检对 A 失效了。
   - 根因：代码只对「非 A 的 vendor」做隔离（`blindHideOther`），当 vendor 是 A 时 `blindHideOther=false`，A 走了「管理员那套：全可见 + 可展开 B/C」的分支。隔离不对称。
2. **拉齐入口错位**：`拉齐 Diff` 按钮只出现在展开的 B 行里，只有能展开 B 行的人（A / 管理员）才点得到；真正的 B 反而看不到、也拉不了。
3. **状态语义混乱**：B 提交后显示 `Submitted (No QC)`，掩盖了「A/B 不一致、待拉齐」这个关键状态；这个状态只在 A 侧的行上冒出来。

## 二、已确定的核心规则

- **R1 盲检对称**：作答阶段 A、B 互盲，各自只能看自己那一格（分数、reason、身份都看不到对方）。
- **R2 提交前盲、拉齐时解盲**：当 A、B 都已提交且结果不一致 → 进入「待拉齐」阶段。此阶段解盲，A、B、管理员都能看到 diff 并执行拉齐。
- **R3 C 只看自己**：QC 复核人 C 只能看到抽样分配给自己的 case，看不到别人的。
- **R4 管理员 / QA 不受限**：始终能看到 A/B/C 全部结果、能分配、能抽样、能拉齐、能导出。

## 三、角色 × 状态 可见性矩阵

维度说明：
- 「看自己」= 看自己那一格的分数 / reason / 状态
- 「看对方」= 看搭档（A看B 或 B看A）的分数
- 「看 C」= 看 QC 复核结果
- 「拉齐」= 能否看到 diff 并执行拉齐
- 空 = 不可见 / 不可操作

### 3.1 供应商 A（vendor.a）

| case 状态 | 看自己(A) | 看对方(B) | 看C | 拉齐 | 可编辑自己 |
|---|---|---|---|---|---|
| 未分配给A | 该 case 不出现在 A 的列表 | | | | |
| A 已分配未提交 | ✓（Assigned） | ✗ | ✗ | ✗ | ✓ Annotate |
| A 已提交、B 未提交 | ✓（等待 B / Submitted） | ✗（盲） | ✗ | ✗ | ✓ 可改 |
| A/B 都提交 · 一致 | ✓ | ✗（仍盲，无需解盲） | ✗ | ✗ | ✓ 可改（定案前） |
| A/B 都提交 · **不一致（待拉齐）** | ✓ | **✓ 解盲** | ✗ | **✓** | ✓ |
| 已抽样进 QC（未定案） | ✓ | 拉齐后一致，可见统一结果 | ✗（C 还没出/不给看） | ✗ | ✓（定案前） |
| QC 定案 Final | ✓（只读） | ✓（统一结果） | ✗（不展示 C 细节给 vendor） | ✗ | ✗ View |

### 3.2 供应商 B（vendor.b）

与 A **完全对称**：把上表的 A/B 对调即可。B 在「待拉齐」阶段同样解盲、同样能拉齐。

### 3.3 供应商 C（vendor.c，QC 复核人）

| case 状态 | 可见 | 可操作 |
|---|---|---|
| 未抽样分配给 C | 不出现在 C 的列表 | |
| 抽中、分配给 C、未复核 | 只看自己这条待复核 case（可参考 A/B 结果——按你选的 C 只看自己复核的，列表里只显示自己那条） | Do QC（白卷作答） |
| C 已复核定案 Final | 自己复核结果（只读） | View |

> 注：C 是否在复核页看到 A/B 作为参考——你选了「C 只看自己复核的」。这里的理解是：**列表页 C 只看到分配给自己的那条**；复核作答页是否显示 A/B 参考，沿用现有 role=C 逻辑（C 需要 baseline 才能 overwrite，作答页保留 A/B 参考是合理的）。如果你要连作答页也不给看 A/B，请标注，我再改。

### 3.4 管理员 / QA（admin / qa.lead / 权限账号）

- 看全部：A / B / C 三格分数、身份、状态、diff。
- 可分配 A、分配 B、抽样指派 C、拉齐 diff、导出全量。
- 权限账号额外：QC 定案后仍可回改 A/B/C。

## 四、状态机（每条 case 的生命周期）

```
Unassigned
  → A Assigned            (分配 A；back-to-back 时同时分配 B)
  → A/B Annotating        (作答阶段，互盲)
  → 都提交:
       一致  → Ready for C Sampling         (可进抽样池)
       不一致 → Pending Reconcile（待拉齐）   (解盲；A/B/管理员可拉齐)
                 → 拉齐后 → Ready for C Sampling
  → Sampled for QC        (抽样指派 C)
  → In C QC               (C 复核中)
  → Final Result Ready    (C 定案)
```

单人评（Single Annotation）：无 B、无拉齐，A 提交后直接 Ready for C Sampling。

## 五、改动点（代码层面）

1. **TaskDetail.tsx 主行渲染**
   - 让盲检隔离对 **A 和 B 对称**：vendor 时，主行永远只渲染「自己的槽位」（含 A）。即把 `blindHideOther` 改为「vendor 且 selfSlot 存在」就隔离，不再对 A 放行。
   - vendor 一律不展开搭档行（`canExpand` 对 vendor 关闭；管理员保留）。
   - 「待拉齐」状态改为在**自己那条主行**上显示（A、B 各自都能看到自己行是「待拉齐」）。
2. **拉齐入口**
   - 从「展开的 B 行」里挪出来，改为：当 `reconcileStatus==="Pending"` 时，主行 Actions 出现「拉齐 Diff」按钮，A、B、管理员都可见可点。
   - `ReconcileModal` 在拉齐阶段展示双方 diff（此时解盲，允许 vendor A/B 看到对方分数）。
3. **状态函数 slotStatus / 过滤器**
   - 待拉齐状态不再只挂 A 槽位；vendor 视角按「自己槽位 + 是否 pending」计算。
4. **C 视角**
   - 列表只显示分配给自己的 case（现有 `caseVisibleTo` 已覆盖），确认主行按 C 槽位渲染、不展开、Do QC / View 正常。
5. **Annotation.tsx**
   - 作答阶段 B 不预填、不显示 A 参考（现有已满足）。
   - 确认 A 作答阶段同样看不到 B（现有 role=A 不显示 B，满足）。

## 六、验证清单

- vendor.a：待拉齐 case 上 A 行显示「待拉齐」+ 有「拉齐 Diff」按钮；A 看不到独立的 B 行；一致/未拉齐时 A 看不到 B 分数。
- vendor.b：与 A 对称，B 行也显示「待拉齐」+ 能拉齐。
- vendor.c：只看到自己复核的 case。
- admin：全部可见、可展开 A/B/C、可拉齐、可抽样。
- `tsc --noEmit` 通过；预览无报错。
